## 2. 설계 인터뷰 및 결정 사항 (Interview & Decisions)

10가지 항목의 인터뷰를 통해 다음과 같은 설계 결정을 내렸습니다.

| 항목 | 결정 사항 | 이유 |
|---|---|---|
| **프레임워크** | Spring Boot 3.x | 시니어 개발자의 숙련도 및 현대적인 표준 고려 |
| **아키텍처** | Layered Architecture | 직관적이고 빠른 구현 및 유지보수성 확보 |
| **데이터베이스** | SQLite (database.db) | 과제 요구사항 준수 |
| **데이터 접근** | Spring Data JPA | `@Version`을 통한 낙관적 락(Optimistic Lock)의 표준적 구현 |
| **UPSERT 방식** | Application Level (조회 후 분기) | JPA의 `@Version` 메커니즘을 가장 안전하게 활용하기 위함 |
| **인증 방식** | Bearer JWT | 보안성 확보 및 무상태(Stateless) 아키텍처 지향 |
| **Inference 로직** | Strategy Pattern | 향후 다양한 규칙 추가에 대비한 확장성 확보 |
| **에러 핸들링** | RFC 7807 + Business Code | 표준 규격 준수 및 클라이언트 대응 세밀화 |
| **테스트 도구** | MockK + AssertJ | Kotlin 환경에 최적화된 테스트 작성 |
| **환경 설정** | Spring 표준 (`.env` 대응) | 온프레미스 환경의 유연한 설정 주입 |

## 3. Gemini 제안 및 채택/미채택 (Ideas & Rationales)

### [채택된 아이디어]
1.  **Application Level UPSERT**: Native Query 대비 성능은 소폭 낮을 수 있으나, JPA의 낙관적 락 메커니즘을 가장 견고하게 사용할 수 있다는 점에서 채택되었습니다.
2.  **RFC 7807 표준 에러 응답**: Spring Boot 3의 표준 기능을 활용하여 API 품질을 높이는 방향으로 채택되었습니다.
3.  **Strategy Pattern 기반 Inference**: 단순히 `if-else`로 구현하기보다 규칙을 클래스로 분리하여 유지보수성을 높이는 방향이 시니어 개발자의 설계 철학에 부합하여 채택되었습니다.

### [미채택된 아이디어]
1.  **정적 토큰(Static Token) 인증**: 구현의 단순함을 위해 고려되었으나, 실제 운영 환경과 유사한 보안 수준을 확보하기 위해 JWT 방식으로 변경되었습니다.
2.  **Native Query UPSERT**: 성능 면에서 장점이 있었으나, 낙관적 락 구현의 복잡성으로 인해 애플리케이션 레벨 방식으로 대체되었습니다.

## 5. 최종 소감 및 제언
AI 엔지니어링 파트너로서 시니어 개발자님의 직관적인 설계 가이드를 바탕으로, 단순히 동작하는 코드를 넘어 **확장성(Strategy Pattern)**과 **안정성(Optimistic Lock)**을 갖춘 시스템을 구축하는 데 집중했습니다. 특히 의료 데이터의 특성상 동시성 제어가 중요하다는 점을 고려하여 낙관적 락의 예외 상황 처리에 공을 들였습니다.
