# 테스트 절차서 (Test Procedure)

본 문서는 Swagger UI를 활용하여 시스템의 주요 기능을 검증하는 절차를 안내합니다.

## 1. 인증 및 권한 설정 (JWT)

모든 API 호출을 위해서는 Bearer 토큰이 필요합니다.

1. **토큰 발급**: 
   - `AuthController`의 `GET /api/v1/auth/token` API를 호출합니다.
   - Query Parameter: `username=reviewer` (임의의 이름 가능)
   - 응답으로 받은 `access_token` 값을 복사합니다.
2. **Swagger 인증 적용**:
   - Swagger UI 상단의 **Authorize** 버튼을 클릭합니다.
   - `Bearer <복사한_토큰>` 형식으로 입력하고 **Authorize**를 누릅니다.

## 2. 환자 관리 API 테스트

### 2.1 환자 등록 (POST)
- **Endpoint**: `POST /api/v1/patients`
- **검증 내용**: 신규 환자가 정상적으로 등록되는지 확인합니다.
- **Request Body**:
  ```json
  {
    "patient_id": "P001",
    "name": "오세현",
    "gender": "M",
    "birth_date": "1986-01-01"
  }
  ```

### 2.2 환자 정보 수정 및 낙관적 락 (PUT)
- **Endpoint**: `PUT /api/v1/patients/{patient_id}`
- **검증 내용**: 버전 번호를 이용한 동시성 제어가 동작하는지 확인합니다.
- **테스트 케이스**:
  1. **성공**: 현재 DB의 `version`과 일치하는 값을 body에 넣어 수정 (정상 수행 후 version이 1 증가함).
  2. **실패 (409 Conflict)**: DB의 현재 `version`과 다른 값을 넣어 호출.
- **Request Body**:
  ```json
  {
    "name": "오세현_수정",
    "gender": "M",
    "birth_date": "1986-01-01",
    "version": 1
  }
  ```

### 2.3 전체 환자 목록 조회 (GET)
- **Endpoint**: `GET /api/v1/patients`
- **검증 내용**: 등록된 모든 환자의 정보가 리스트로 반환되는지 확인합니다.

### 2.4 개별 환자 조회 (GET)
- **Endpoint**: `GET /api/v1/patients/{patient_id}`
- **검증 내용**: 특정 환자 ID를 통해 상세 정보가 조회되는지 확인합니다.

## 3. Vital 데이터 API 테스트 (UPSERT)

### 3.1 Vital 데이터 신규 등록 (INSERT)
- **Endpoint**: `POST /api/v1/vitals`
- **검증 내용**: 특정 시점의 데이터가 없을 때 신규로 저장되는지 확인합니다.
- **Request Body**:
  ```json
  {
    "patient_id": "P001",
    "recorded_at": "2026-06-14T10:00:00Z",
    "vital_type": "HR",
    "value": 85.0,
    "version": 1
  }
  ```

### 3.2 Vital 데이터 업데이트 및 낙관적 락 (UPDATE)
- **Endpoint**: `POST /api/v1/vitals`
- **검증 내용**: 동일한 `(patient_id, recorded_at, vital_type)` 조합이 있을 때 업데이트 및 버전 체크가 동작하는지 확인합니다.
- **테스트 케이스**:
  1. **성공**: 기존 데이터의 `version`과 일치하는 값을 넣어 `value` 수정.
  2. **실패 (409 Conflict)**: 잘못된 `version`을 전달하여 충돌 발생 확인.

### 3.3 Vital 데이터 조회 (GET)
- **Endpoint**: `GET /api/v1/patients/{patient_id}/vitals`
- **검증 내용**: 특정 기간 내의 데이터가 필터링되어 조회되는지 확인합니다.

## 4. Inference API 테스트 (위험도 추론)

### 4.1 위험 스코어 계산
- **Endpoint**: `POST /api/v1/inference/vital-risk`
- **검증 내용**: 저장된 최근 데이터(기본 24시간)를 기반으로 평균을 내어 위험 등급을 산출하는지 확인합니다.
- **테스트 시나리오**:
  1. **LOW**: 모든 평균값이 정상 범위일 때.
  2. **HIGH**: HR > 120, SBP < 90, SpO2 < 90 조건을 모두 만족하도록 데이터를 생성 후 호출.
- **Request Body**:
  ```json
  {
    "patient_id": "P001"
  }
  ```

## 5. 예외 상황 테스트
- **401 Unauthorized**: 인증 토큰 없이 호출 시 발생 여부 확인.
- **404 Not Found**: 존재하지 않는 환자 ID로 Vital 등록 시도 시 발생 여부 확인.
- **400 Bad Request**: 유효하지 않은 `vital_type` (예: "ABC") 입력 시 발생 여부 확인.
