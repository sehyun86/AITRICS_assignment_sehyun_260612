# 시스템 설계 (System Design)

이 문서는 AITRICS BE Senior 채용 과제를 위한 시스템 설계 명세서입니다. 14년 8개월 경력의 시니어 개발자의 가이드와 인터뷰를 통해 결정된 아키텍처 및 기술 스택을 기술합니다.

## 1. 기술 스택 (Technology Stack)

- **Language**: Kotlin
- **Framework**: Spring Boot 3.x
- **Database**: SQLite (파일명: `database.db`)
- **Persistence**: Spring Data JPA
- **Security**: Spring Security + JWT (JSON Web Token)
- **API Documentation**: Springdoc-openapi (Swagger UI)
- **Testing**: JUnit 5, MockK, AssertJ, MockMvc
- **Build Tool**: Gradle (Kotlin DSL)
- **Containerization**: Docker (Dockerfile 제공)

## 2. 아키텍처 구조 (Architecture)

직관적이고 유지보수가 용이한 **Layered Architecture**를 채택합니다.

- **Controller (Interface Layer)**: API 엔드포인트 정의, 요청 유효성 검사, JWT 인증 처리.
- **Service (Application Layer)**: 비즈니스 로직 수행, 트랜잭션 관리, Inference 규칙 적용 (Strategy Pattern).
- **Repository (Infrastructure Layer)**: DB 접근 (Spring Data JPA).
- **Domain (Domain Layer)**: 엔티티 및 핵심 도메인 로직 (Entity, DTO, Constants).

## 3. DB 스키마 설계 (Database Schema)

### Patients (환자 정보)
| Column | Type | Constraints | Description |
|---|---|---|---|
| patient_id | TEXT | PRIMARY KEY | 환자 고유 식별자 |
| name | TEXT | NOT NULL | 성명 |
| gender | TEXT | NOT NULL | 성별 (M/F) |
| birth_date | DATE | NOT NULL | 생년월일 |
| version | INTEGER | NOT NULL | 낙관적 락을 위한 버전 |

### Vitals (생체징후 데이터)
| Column | Type | Constraints | Description |
|---|---|---|---|
| id | INTEGER | PRIMARY KEY AUTOINCREMENT | PK |
| patient_id | TEXT | NOT NULL, FK(Patients) | 환자 ID |
| recorded_at | DATETIME | NOT NULL | 측정 시간 |
| vital_type | TEXT | NOT NULL | HR, RR, SBP, DBP, SpO2, BT |
| value | REAL | NOT NULL | 측정값 |
| version | INTEGER | NOT NULL | 낙관적 락을 위한 버전 |

- **Index**: `UNIQUE INDEX idx_vital_composite (patient_id, recorded_at, vital_type)` 적용하여 UPSERT 식별자로 활용.

## 4. 핵심 설계 (Key Design Decisions)

### (1) Optimistic Lock (낙관적 락)
- JPA의 `@Version` 어노테이션을 사용하여 구현합니다.
- **환자 수정 API**: `PUT /api/v1/patients/{id}` 호출 시 Request Body의 `version`과 DB의 `version`을 비교합니다.
- **Vital UPSERT**:
    - `patient_id`, `recorded_at`, `vital_type`으로 조회.
    - 존재하지 않으면 `version = 1`로 신규 저장.
    - 존재하면 기존 엔티티에 새 `value`와 요청받은 `version`을 세팅하여 `save()` 호출 -> JPA가 `version` 불일치 시 `ObjectOptimisticLockingFailureException` 발생.

### (2) Inference Engine (Strategy Pattern)
- `InferenceRule` 인터페이스를 정의하고, 각 Vital 위험 조건(HR, SBP, SpO2)을 클래스로 분리합니다.
- `InferenceService`는 등록된 모든 규칙을 순회하며 스코어를 계산하여 확장성을 확보합니다.

### (3) 인증 (Authentication)
- Bearer JWT 방식을 사용합니다.
- 환경변수(`JWT_SECRET`)로 관리되는 토큰을 검증합니다.
- Spring Security Filter에서 유효성을 확인하고 인증 객체를 생성합니다.

### (4) 에러 핸들링 (Error Handling)
- **RFC 7807 (Problem Details)** 표준 규격 사용.
- 비즈니스 에러 코드 정의 (예: `PATIENT_NOT_FOUND`, `VERSION_CONFLICT`).
- `@RestControllerAdvice`에서 전역 처리.

## 5. 디렉토리 구조 (Project Structure)

```text
src/main/kotlin/com/aitrics/assignment/
├── auth/               # JWT, Security Config
├── common/             # Base Entity, Global Exception, Constant
├── controller/         # API Controllers, DTOs
├── service/            # Business Logic
│   └── inference/      # Rule Engine Strategies
├── repository/         # JPA Repositories
└── domain/             # Entities
```

## 6. 비기능적 요구사항 구현 계획

- **Dockerfile**: Multi-stage build를 사용하여 최적화된 이미지 제공.
- **Config**: `application.yml`에 기본값을 설정하고, 환경변수(`.env` 대응 가능)로 런타임 설정 주입.
- **Swagger**: `/swagger-ui/index.html`을 통해 상세한 API 명세와 예시 제공.
- **Test**: 예외 케이스(락 충돌, 권한 없음, 데이터 없음 등)를 포함한 통합 테스트 시나리오 작성.
